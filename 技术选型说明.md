# 技术选型说明

物品动态装箱展示系统 —— 前后端技术选型与理由。

## 一、关键设计决策

| 决策点 | 结论 |
| --- | --- |
| 可视化形式 | 真 3D 渲染，展示物品在箱内的实际码放 |
| 装箱算法 | 前后端混合：余量充足时前端启发式即时摆放；启发式判断放不下时，整批交后端求最优（近优）解 |
| 后端 | 需要完整后端，含商品管理后台（录入、库存、上下架） |
| 库存校验 | 仅在最终确认清单时做一次性校验，装箱过程不实时判断 |
| 导出 | 仅导出商品类型 + 数量的纯文本清单（前端生成即可） |

## 二、前端选型

### 装箱应用 (packing-web)

| 技术 | 用途 | 选择理由 |
| --- | --- | --- |
| React + TypeScript + Vite | 应用基座 | 几何/数据密集场景下 TS 省调试成本；Vite 启动与构建快 |
| react-three-fiber + drei | 3D 渲染 | 声明式写 Three.js，状态驱动自动重渲染，远比命令式好维护；drei 提供相机控制、辅助网格等现成件 |
| `<Instances>` 实例化渲染 | 性能 | 商品数量上去后合批渲染同类网格，避免帧率下滑 |
| react-spring | 过渡动画 | 后端返回新布局时，物品位置平滑过渡，直观展示"重新码放" |
| dnd-kit | 拖拽 | 现代 React 拖拽方案；DOM 层完成拖拽，drop 只触发"加入商品"，摆放交给算法，避开 DOM↔WebGL 坐标同步难题 |
| Zustand | 状态管理 | 轻量，与 R3F 配合极好；箱内容、占用率、清单共享一份 store |
| Web Worker | 跑启发式算法 | 即时计算摆放不阻塞 UI 线程 |
| Tailwind CSS | 常规 UI | 列表、清单面板等 2D 界面 |

> 备选：Vue 栈可用 TresJS（Vue 版 R3F），但 3D 生态、案例与踩坑记录明显少于 R3F，故优先 React。

### 管理后台 (admin-web)

| 技术 | 用途 | 选择理由 |
| --- | --- | --- |
| React + Ant Design | CRUD 后台 | 商品录入/库存/上下架是表格表单重活，AntD 现成组件大幅省工 |

可与装箱端共用一个 React 工程不同路由，也可拆为独立应用，按部署偏好决定。

## 三、算法策略

- **即时层（前端）**：TS 实现启发式装箱（Extreme Point / Deepest-Bottom-Left-with-Fill / First-Fit-Decreasing 之类），跑在 Web Worker，每加一件增量计算摆放与占用率。
- **优化层（后端）**：触发条件为启发式判断放不下。后端对"已选全部商品 + 新商品"整批求解，时间受限（2~3s）返回近优布局。
- **"最优"的真实含义**：3D 装箱 NP-hard，真正最优仅小规模可解。实际用时间受限的构造启发式 + 局部搜索 / 元启发式（模拟退火、遗传），返回时限内最优结果。触发点在"快满时"，规模有限，足够。
- **是否允许物品旋转 / 是否做重力物理堆叠**：旋转每件最多 6 种轴对齐朝向，是算法与 UI 的复杂度乘数，建议先确认需求；纯几何摆放通常够用，物理堆叠（rapier/cannon-es）更真实但维护成本陡增，建议初期不做。

## 四、后端选型

**结论：FastAPI 模块化单体起步，把求解器隔离成可独立拆分的内部模块。**

| 技术 | 用途 | 选择理由 |
| --- | --- | --- |
| FastAPI (Python) | 单一后端 | CRUD 用 SQLAlchemy + Alembic 成熟；求解器本就是 Python 强项（py3dbp 等），同语言同进程省一次网络跳转与第二门语言维护成本；自带 OpenAPI 文档、异步性能好 |
| 模块划分 | 可维护性 | `auth` / `catalog` / `solver` / `manifest` 分层；求解器藏在清晰函数接口后，将来计算量大了可原样抽成独立微服务，几乎不改业务代码 |
| 进程池 / Celery worker | 跑求解 | 重计算不堵塞 API worker；v1 同步请求 + 硬超时 + loading 态，将来可升级为异步 job + WebSocket 推送 |
| PostgreSQL | 持久化 | 商品、库存、装箱记录 |
| 对象存储 S3 / MinIO + CDN | 缩略图 | 图不入库 |
| Redis | 缓存 / 队列 | 按"商品集合+箱体"缓存求解结果；可选作 Celery broker |
| Docker / docker-compose | 部署 | 容器化，环境一致 |

> 备选：若团队更熟 TS，可用 NestJS 做主应用 + 独立 Python 求解微服务，但引入两门语言，维护面更大。

## 五、共享契约（最关键的可维护性约定）

前端 TS 启发式与后端 Python 求解器必须实现**同一份摆放 schema**：统一单位 (mm)、坐标系（原点在箱底角，x 长 / y 宽 / z 高向上）、placement 结构（productId、position 最小角坐标、rotationType 0..5、旋转后 footprint）、箱体尺寸定义。两端布局可互换——前端摆的后端能接着算，后端算的前端能直接渲染。详见 `packing-contract.ts` 与 `packing_contract.py`，两份保持同步。

## 六、选型一览

```
装箱前端  React + TS + Vite + react-three-fiber/drei (Instances + react-spring)
          + dnd-kit + Zustand + Tailwind；启发式装箱在 Web Worker；纯文本清单前端导出
管理后台  React + Ant Design
后端      FastAPI 模块化单体 (auth / catalog / solver / manifest)
          求解器在进程池或 Celery worker、时间受限
          PostgreSQL + 对象存储(S3/MinIO) + Redis；Docker 容器化
契约      前后端共享 placement schema（坐标系 / 单位 / 结构 / 箱体尺寸）
```

## 七、风险与待定项

- **NP-hard 张力**：把"即时反馈"与"尽量装满"拆成前端启发式 + 后端时间受限求解两层，向产品方明确"最优=近优"。
- **DOM↔3D 拖拽**：拖拽只在 DOM 层完成，摆放交算法，绕开坐标同步坑。
- **布局重排体验**：后端返回的布局与前端不同，用过渡动画平滑展示。
- **待定业务开关**：确认成功后是否扣减/预占库存（默认仅记录不扣减）；是否允许物品旋转；是否需要物理堆叠。
